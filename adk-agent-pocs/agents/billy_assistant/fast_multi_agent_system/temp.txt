Review criteria

Latency
	•	How many LLM calls are needed for the common path?
	•	Does routing happen quickly and deterministically where possible?
	•	Are unnecessary handoffs, re-planning steps, or repeated clarifications avoided?
	•	Can simple requests be answered by one expert directly?

Maintainability
	•	Are responsibilities clearly separated between router, experts, and tools?
	•	Is routing logic simple enough to understand and change safely?
	•	Are prompts, tools, and policies modular rather than duplicated across agents?
	•	Can behavior be tested with predictable inputs and outputs?

Adding new experts
	•	How much work is needed to add a new expert end-to-end?
	•	Is there a clear contract for an expert: scope, input, output, tools, escalation rules?
	•	Can a new expert be added without changing many existing agents?
	•	Does the router scale cleanly as the number of experts grows?

Readability
	•	Is it easy to understand which agent does what?
	•	Are names, boundaries, and delegation paths explicit?
	•	Can an engineer quickly trace how a user request flows through the system?
	•	Are prompts/instructions concise and not overloaded with mixed responsibilities?

Important extra criteria
	•	Correct routing: Does the request reach the right expert reliably?
	•	Fallback behavior: What happens when no expert matches or confidence is low?
	•	Safety and control: Are sensitive actions guarded with confirmation and clear ownership?
	•	Observability: Can you see routing decisions, latency per step, tool calls, and failures?
	•	Testability: Can you evaluate router accuracy, expert quality, and regression over time?

Findings

High: Expert-to-expert transfer is specified in prompts but not clearly wired in the agent graph. The invoice prompt tells invoice_agent to transfer to support on how-to requests at invoice_agent.txt:35-36, and the support prompt says the inverse for explicit data requests at support_agent.txt:13-15. But the direct experts are built by cloning the template and injecting tools only at expert_registry.py:58-60. The only explicit sub_agents wiring I found is on the top-level router at agent.py:36-41. Unless ADK allows cross-app transfers without sibling wiring, those prompt instructions are non-executable or at least unproven.

Medium: The eval harness does not actually score the routing signal it says is primary, and it tolerates extra work. The documented metric is only tool_trajectory_avg_score with ANY_ORDER, which explicitly tolerates extra calls at README.md:35. The same README says routing correctness is really inferred from intermediate_responses, but “no built-in metric currently scores it” at README.md:52. With only six cases total at README.md:17, this will miss both routing regressions and latency regressions.

Medium: Adding a new expert is not as localized as the registry comments imply. The registry advertises itself as the single source of truth and says “everything else wires itself automatically” at expert_registry.py:2-10, but in practice you still need to update the router’s hardcoded agent list at agent.py:36-41, maintain import order in __init__.py:2-17, and edit prompt-level routing policy in router_agent.txt:5-26 and receptionist_agent.txt:19-31. That is workable for two experts, but it will drift as the expert count grows.

Low: There is a dead reroute mechanism that increases conceptual load without affecting behavior. request_reroute and PUBLIC_ROUTING_ESCALATION are defined at context_tools.py:10-13 and context_tools.py:55-64, and get_conversation_context exposes routing_escalation at context_tools.py:33, but I did not find routing logic or prompts consuming it. That suggests an abandoned design path and makes the real ownership model harder to read.

Assessment
Latency is decent for a prompt-routed prototype, but not minimal. The common single-domain path is effectively 2 LLM calls: the lightweight router, then the selected expert. The router is optimized for speed with a lite model, static prompt, and include_contents="none" at agent.py:25-43. Multi-domain requests are effectively 4 LLM calls: router, orchestrator, then one helper call per expert involved, as wired at orchestrator_agent.py:27-30 and expert_registry.py:64-69. Simple requests cannot be answered by one expert directly because the root is always a pure classifier at agent.py:20-34.

Maintainability and readability are mixed. The expert registry is a real improvement: it centralizes prompt injection, shared tools, and helper generation at expert_registry.py:58-69. But the routing contract is duplicated across the router, receptionist, and expert prompts, so behavior is spread across several natural-language files instead of one enforceable policy. That hurts change safety.

Correct fallback and safety are better than observability and testability. Fallback is clear: uncertain cases go to the receptionist at router_agent.txt:49, and the receptionist handles out-of-scope requests gracefully at receptionist_agent.txt:14-15. Safety is solid for invoice writes: the prompt requires confirmation at invoice_agent.txt:39-40, and the tool layer enforces it at invoice_tools.py:66-79. Observability is thin: debug is opt-in via an env flag at agent.py:15, and the only hooks I found are stderr callbacks in debug.py:32-44, with no structured routing events or latency capture.
