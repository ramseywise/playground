How can a2ui be used to complete an existing react page and an agent?

Using **A2UI** (Agent-to-User Interface) allows you to bridge the gap between your AI agent's logic and your React frontend. Instead of the agent generating unpredictable code, it sends a declarative **JSON blueprint** that your React app renders using your own trusted components.

Here is how to use it to complete an existing React page and integrate it with your agent.

---

## 1. Setup the React Frontend
To integrate A2UI into an existing React page, you need the official renderer package.

### Installation
```bash
npm install @a2ui-sdk/react @a2ui/web-lib
```

### Implementation
Wrap your application (or specific page section) with the `A2UIProvider`. This manages the state of the UI messages and the data model.

```tsx
import { A2UIProvider, A2UIRenderer, standardCatalog } from '@a2ui-sdk/react';

// 1. (Optional) Extend the catalog with your existing React components
const myCustomCatalog = {
  ...standardCatalog,
  components: {
    ...standardCatalog.components,
    // Map an A2UI key to your actual React component
    MyCustomButton: (props) => <button className="btn-primary" {...props} />
  }
};

function MyAgentPage() {
  const [messages, setMessages] = useState([]);

  // This function sends user actions back to your backend agent
  const handleAction = async (action) => {
    const response = await fetch('/api/agent', {
      method: 'POST',
      body: JSON.stringify(action),
    });
    const nextMessage = await response.json();
    setMessages((prev) => [...prev, nextMessage]);
  };

  return (
    <A2UIProvider catalog={myCustomCatalog} messages={messages}>
      <div className="page-layout">
        <h1>Agent Interaction</h1>
        {/* The renderer displays the UI generated by the agent */}
        <A2UIRenderer onAction={handleAction} />
      </div>
    </A2UIProvider>
  );
}
```

---

## 2. Setup the Agent (Backend)
The agent's job is to output the **A2UI JSON format** instead of plain text. This JSON describes the structure (layout) and the data (model).

### Example Agent Output (JSON)
Your agent (e.g., via LangChain, Gemini, or a custom Node.js/Python script) should return a payload like this:

```json
{
  "template": {
    "root": "main_layout",
    "components": {
      "main_layout": {
        "Column": {
          "children": ["title", "input_field", "submit_btn"]
        }
      },
      "title": { "Text": { "text": "Enter your shipping details" } },
      "input_field": { 
        "TextField": { 
          "label": "Full Name",
          "text": { "path": "/user/name" } 
        } 
      },
      "submit_btn": { 
        "Button": { 
          "label": "Confirm",
          "action": { "type": "submit", "id": "confirm_order" } 
        } 
      }
    }
  },
  "data": { "user": { "name": "" } }
}
```

---

## 3. Key Concepts to Complete the Loop

### Data Binding
A2UI uses **JSON Pointers** (e.g., `/user/name`). 
* When the user types in the React `TextField`, it updates the local state in the `A2UIProvider`.
* The agent only sees the data when the user triggers an `action` (like clicking "Confirm"). This keeps the interaction secure and efficient.

### Progressive Completion
You don't have to replace your whole page. You can use `A2UIRenderer` as a **widget** inside an existing layout. 
* **Existing Page:** Keeps your header, sidebar, and static content.
* **A2UI Agent:** Dynamically generates the "Work Area" or "Chat Bubble" based on the task (e.g., switching from a search form to a data table automatically).

### Security
Because the agent sends **declarative JSON** and not `eval()`-able JavaScript, it cannot perform XSS attacks. It can only "request" components that you have explicitly allowed in your `catalog`.

---

### Summary Checklist
1.  **Define Catalog:** Map your React components to A2UI component names.
2.  **Provider/Renderer:** Add `<A2UIProvider>` and `<A2UIRenderer>` to your React page.
3.  **Prompt the Agent:** Instruct your LLM to "respond using A2UI JSON schema" for any task requiring a UI.
4.  **Handle Actions:** Set up a POST endpoint to receive user inputs from the UI and feed them back into the agent's next turn.

So an agent + a2ui can support existing pages, so the agent can update fields when chatting and also read the content

Exactly. You’ve hit on the core "magic" of A2UI. It isn’t just a one-way display; it creates a **bidirectional state loop** between your existing page and the agent.

Because A2UI uses a shared **Data Model** (a JSON state tree) that both the React frontend and the Agent can access, they can stay in perfect sync.

### 1. How the Agent "Reads" Content
The agent doesn't "scrape" your DOM. Instead, it reads the **Data Model**. 
* **The Shared State:** When you initialize an A2UI surface on your page, you define a JSON state (e.g., `{ "user": { "email": "test@me.com" } }`).
* **The "Context" Injection:** When a user interacts with the page (like clicking a "Help me fill this out" button), the A2UI renderer can package the current values of those fields and send them back to the agent as part of the action payload.
* **Stateless Sync:** Newer versions of A2UI support "Data Model Sync," which automatically attaches the latest values of all bound fields to every message the user sends to the agent.



---

### 2. How the Agent "Updates" Fields
When the agent wants to fill a field for the user, it doesn't write code; it sends a **Data Model Update** message.

* **Reactive Binding:** In your React code, your input field is "bound" to a specific path (e.g., `/shipping/zipCode`).
* **The Update Message:** The agent sends a JSON snippet: 
  ```json
  { "op": "updateDataModel", "path": "/shipping/zipCode", "value": "90210" }
  ```
* **Instant UI Refresh:** The `A2UIProvider` in your React app sees this update and immediately re-renders the specific component bound to that path. To the user, it looks like the agent "typed" into the form.

---

### 3. Integration with Your Existing Page
You don't have to turn your whole page into an "A2UI page." You can use it for **Hybrid Automation**:

| Feature | How it works in A2UI |
| :--- | :--- |
| **Field Auto-fill** | Agent sends `updateDataModel` to fill out a complex form based on a user's chat message ("Set my address to 123 Main St"). |
| **Validation** | The agent can read the current form state, see a mistake, and send an `updateComponents` message to add an "Error" component or highlight a field in red. |
| **Contextual UI** | If the user mentions "I'm having trouble with the pricing," the agent can dynamically inject a "Price Calculator" widget into your existing layout. |

### Summary of the "Loop"
1. **React State** → User types in a bound `TextField`.
2. **Action Event** → User clicks a button; the current **Data Model** is sent to the **Agent**.
3. **Agent Logic** → Agent processes the data and decides what needs to change.
4. **JSON Response** → Agent sends back a **Data Model Update** or a new **Component Blueprint**.
5. **React Update** → The UI updates instantly to reflect the Agent's changes.

This makes the agent feel like a "co-pilot" that actually has its hands on the controls of your application, rather than just a chatbot sitting in a corner.

If a user for example click create customer, then it use a rest API for creation. Can it at the same time send that event back to the agent so it know that a customer has been created. No need for answering back to the user.

Spot on. You don't always need to burn tokens (or wait for a "thought") just to sync data. You can treat the agent's **Session State** like a client-side store that only involves the LLM when logic is actually required.

There are two primary ways to update the agent state without a full LLM inference cycle:

### 1. Direct State Injection (The "Fast Path")
When your React app completes an action (like that REST call), you can send the data directly to the **Agent's Orchestrator** (the backend code wrapping the LLM). 

* **How it works:** Your backend receives the event, updates the session's "Context" or "Short-term Memory" database (like Redis or a local state object), and returns a `204 No Content` or a simple `{ "status": "synced" }`.
* **The Result:** The next time the user *does* talk to the agent, the updated state is already sitting in the prompt's "System Message" or "Context Window."

### 2. A2UI "Data Model Only" Updates
A2UI differentiates between a **UI Update** (rendering new buttons) and a **Data Update** (changing values).

If your React app updates a field that is bound to the A2UI Data Model:
* The `A2UIProvider` updates the local JSON tree.
* If you have **Auto-Sync** enabled, this JSON is mirrored to the backend.
* **Zero LLM cost:** The backend simply stores the new JSON. The LLM is never woken up until the user hits "Send" or an action triggers a "reasoning" step.

---

### Why this is a game changer:

| Scenario | Traditional Agent | A2UI + State Sync |
| :--- | :--- | :--- |
| **User types an address** | Agent is blind until the user submits. | Agent's "Context" updates in real-time as the user types. |
| **API success event** | Agent has no idea unless it "polls" the DB. | React pushes a silent event; Agent "knows" instantly. |
| **Cost / Latency** | High (Calling LLM for every update). | **Zero** (It's just a standard JSON POST). |

### Example Workflow:
1.  **User:** Clicks "Create Customer."
2.  **React:** Calls `/api/create`.
3.  **React:** Calls `a2ui.syncState({ lastCreatedId: 123 })`.
4.  **Backend:** Saves `{ lastCreatedId: 123 }` to the session. **(No LLM involved yet)**.
5.  **User:** Later asks "What was that ID?"
6.  **Backend:** Sends the prompt to the LLM: *"User just asked a question. Context: lastCreatedId is 123."*
7.  **LLM:** "The ID was 123!"

This keeps your agent **grounded in reality** without the lag or cost of constantly "asking the brain" to watch the UI.
