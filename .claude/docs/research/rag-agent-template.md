## 1. Multi-Turn Conversation

### The core problem
Single-turn RAG pipelines can't handle pronouns, follow-ups, or topic continuations.
Multi-turn requires resolving "What if that doesn't work?" against prior context.

### Approaches

| Approach | Tradeoff |
|---|---|
| **Full history in context** | Simple, degrades at ~10 turns |
| **Summarized history** | Adds latency, loses detail |
| **History-aware query rewriting** | Best retrieval quality — small Haiku call before retrieve |
| **Retrieval-augmented memory** | Cross-session recall — Phase 2 feature |

### Implemented pattern (help-assistant + listen-wiseer)
`rewrite_query` node fires when coreference signals detected (`" it "`, `" they "`, `"the artist"`).
Resolves last 5 turns into a standalone question before retrieval. Single Haiku call, ~200ms.
listen-wiseer does it rule-triggered; help-assistant always rewrites. Both work.

### LangGraph memory
- `MemorySaver` — in-process, good for tests and single-instance
- `AsyncRedisSaver` (TTL 24h) — required for Fargate / multi-instance prod (help-assistant Phase C)
- `BaseStore` with namespaced vectors — episodic memory across sessions (listen-wiseer Phase 4b)
- Template: `MemorySaver` default, `checkpointer` param for injection

## 2. RAG Retrieval Strategies Evaluated

### 2.1 Chunking strategies (all implemented in help-assistant `src/preprocessing/`)

| Strategy | Mechanism | Verdict |
|---|---|---|
| `html_aware` | Heading-boundary split → recursive fallback | **Primary** — best for structured help docs |
| `fixed` | Hard word splits, no overlap | Baseline benchmark only |
| `overlapping` | Fixed + configurable overlap | Better recall, more redundant embeddings |
| `structured` | `\n\n → \n → sentence → word` recursion | Good plain-text fallback |
| `adjacency` | Fixed + positional IDs for `neighbors(chunk_id)` | Graph/context expansion at query time |
| `parent_doc` | Small child chunks indexed, parent returned | Best precision + context; needs two-level index |
| `section_aware` | Consumes `doc["sections"]` from scraper | **Future** — higher fidelity, blocked on text/full_text fix |
| `proposition` | LLM rewrites each fact as standalone | Skip — LLM per chunk at ingest is too expensive |
| `semantic` (regex) | Sentence boundary split | Skip — fragile for non-English |

**Key findings:**
- Scraper outputs `full_text` (plain text, HTML stripped), not `text` — chunker must handle both keys
- `HtmlAwareChunker` re-detects headings from `full_text` via regex; a `SectionAwareChunker`
  consuming `doc["sections"]` directly would be more reliable (that field has `heading`, `level`, `content`)
- `min_tokens=50` (default) drops very short section bodies — tests need multi-sentence content

### 2.2 Retrieval strategies

| Strategy | How | Verdict |
|---|---|---|
| **Hybrid BM25 + k-NN** (OpenSearch) | Server-side score fusion, weights configurable | **Current default** — better than Python-side merging |
| **Dense only** (k-NN) | Pure vector similarity | Lower recall on keyword-heavy queries |
| **Sparse only** (BM25) | TF-IDF bag-of-words | Poor on semantic/paraphrase queries |
| **Multi-query (RAG-Fusion)** | N query variants → retrieve for each → deduplicate | +10–15% recall, implemented as `query_variants` in state |
| **HyDE** | Generate hypothetical answer → embed that | Good for factual queries; adds 1 LLM call |
| **Step-back** | Abstract to broader question first | Good for procedural; not yet evaluated |
| **A-RAG tool routing** | Keyword / semantic / chunk-level as separate LLM tools | +5–13% QA accuracy; adds agentic overhead |

**Current implementation (help-assistant):**
- `hybrid_search(query_text, query_vector, k=5, bm25_weight=0.3, vector_weight=0.7)`
- 3 query variants generated by `QueryAnalyzer`, deduped by chunk.id
- Weights are hardcoded — need eval to tune

**Embedder:**
- `intfloat/multilingual-e5-large` (1024-dim, ~2.2GB, local via sentence-transformers)
- **E5 prefix rule** is critical: `"query: "` for queries, `"passage: "` for docs
- This rule must live in the abstract `Embedder` protocol, not the implementation, so it can't be bypassed

### 2.3 CRAG (Corrective RAG) — implemented

After retrieval, a Haiku grader judges each chunk's relevance (score ≥ 0.5 = relevant).
- All relevant → generate
- Some relevant → filter + generate
- None relevant → rewrite + retry (max 1 retry, `retry_count >= 2` signals stop)

**Gotcha:** `grade_docs` increments `retry_count` before returning (so first run = count 1).
`check_sufficient` allows retry when `retry_count < 2`, so exactly one actual retry happens.
This is counterintuitive — document clearly.

## 3. Reranker — Strategies and Decision

### 3.1 Options evaluated

| Option | Latency | Quality | Ops cost | GDPR |
|---|---|---|---|---|
| **LLM listwise** (Haiku prompt ranking) | 400–800ms | High | Extra LLM call | OK |
| **Cross-encoder** (MiniLM-L-6-v2, ~22MB) | 50–100ms | High | Load at startup | OK |
| **Cohere Rerank API** | 100–300ms + network | Highest | Paid API | ❌ data leaves infra |
| **ColBERT** (late interaction) | ~20ms with index | Very high | PyColBERT/Vespa — heavy | OK |

### 3.2 Decision: provide both LLM + cross-encoder, configurable

**Why both:**
- Want to A/B test them against each other on the eval suite before committing
- Cross-encoder is already a `sentence-transformers` dep (same dep as embedder)
- LLM listwise costs ~$0.001/query at Haiku — acceptable for experimentation
- Neither should be hardcoded; swap via Registry or `reranker_strategy` config

**Implementation shape:**
```python
class Reranker(Protocol):
    def rerank(self, query: str, chunks: list[GradedChunk], top_k: int) -> list[RankedChunk]: ...

class CrossEncoderReranker:  # sentence-transformers, no API
    model = "cross-encoder/ms-marco-MiniLM-L-6-v2"

class LLMListwiseReranker:   # Haiku prompt, returns ordered list
    ...
```

**n-candidates rule of thumb:** retrieve k=10 → CRAG grade → 3–6 relevant → rerank → top 3 for generation.
Too many into reranker → latency spike. Too few → no gain over CRAG alone.

**Confidence score:** reranker should output `confidence_score: float` per chunk (cross-encoder
gives a relevance logit; LLM can output a 0–1 score). This flows to `confidence_gate`.

## 4. Intent Classification and Planning

### 4.1 Current: rule-based `QueryAnalyzer`

`QueryAnalyzer` in `src/orchestration/query_understanding.py` (458 lines, help-assistant):
- Keyword/regex matching → `{factual, procedural, exploratory, troubleshooting}`
- `TERM_EXPANSIONS` dictionary for domain terminology
- Entity extraction (ticket IDs, invoice numbers, dates, amounts)
- Query decomposition on "og" (Danish "and") and multiple `?`
- Complexity scoring: `simple/moderate/complex`

Identical module reused in listen-wiseer (`src/rag_core/orchestration/query_understanding.py`).

**Where it breaks:**
- Ambiguous queries that don't contain exact keywords
- Multi-intent queries ("I want to cancel but first export my data")
- Cross-lingual queries (keyword lists are language-specific)

### 4.2 Alternative: LLM-based planning (structured output)

```python
class QueryPlan(BaseModel):
    intent: Literal["how_to", "troubleshoot", "reference", "chit_chat", "out_of_scope"]
    tools_to_use: list[Literal["rag", "snippets_db", "direct"]]
    needs_clarification: bool
    clarification_question: str | None
    confidence: float

plan = await haiku.with_structured_output(QueryPlan).ainvoke(query)
```

- ~200ms, ~$0.001/query
- Handles ambiguous / multi-intent naturally
- Enables dynamic tool routing without code changes when new tools added

### 4.3 Decision: rule-based default, LLM optional via `planning_mode` config

The template ships rule-based (zero cost, proven). `planning_mode: "llm"` activates structured
output planning. The `QueryRouter` class (already in help-assistant) is the right home for this.

## 5. Is It Multi-Agent? The Architecture Question

### 5.1 What listen-wiseer and help-assistant actually are

**listen-wiseer** (`src/agent/graph.py`): single compiled ReAct graph.
- Nodes: `trim_history → classify_intent → clarify_or_proceed | rewrite_query → agent → call_tools → validate_tool_output → agent (loop)`
- One shared `AgentState`, one LLM with tools bound
- "Multi-agent" only in the sense that it uses tools — tools are functions, not agents

**help-assistant** (`src/orchestration/graph.py`): single compiled deterministic pipeline.
- Nodes: `classify_intent → rewrite_query → retrieve → grade_docs → generate → confidence_gate`
- No tools, no loops (except CRAG retry), no parallel execution

Neither is multi-agent by the strict definition: **separately deployed agents with their own state,
goals, and communication protocol.**

### 5.2 What makes something truly multi-agent in LangGraph

LangGraph's supervisor pattern (official docs):

```
supervisor_graph:
    supervisor_node  ← decides which subgraph to invoke next
        ↓  (Command routing)
    retrieval_subgraph   ← compiled independently, own state
    reranker_subgraph    ← compiled independently, own state
    generation_subgraph  ← compiled independently, own state
```

Each subgraph:
- Has its own `StateGraph` and compiled `CompiledStateGraph`
- Communicates with the supervisor via `Command(goto=..., update={...})`
- Can be tested, swapped, and deployed independently
- Has its own tools/LLM — the supervisor doesn't share these

This is different from "nodes that call an LLM" — the modularity is structural, not just
conceptual.

### 5.3 Why go multi-agent for the RAG template

The architecture slides show Planning → Retrieval → Reranking → Generation as distinct agents
with their own responsibilities. Making them actual subgraphs gives:

1. **Independent testability**: test the retrieval subgraph without the generation subgraph
2. **Swappability**: swap the reranker subgraph (LLM vs cross-encoder) without touching generation
3. **Parallelism**: supervisor can invoke retrieval + context-augmentation in parallel (future)
4. **Observability**: each subgraph appears as a separate span in LangFuse traces
5. **True modularity**: the template demonstrates a production pattern, not just a pipeline

### 5.4 Supervisor graph design for the template

```
SupervisorState
    ├── messages (add_messages)
    ├── query, standalone_query, intent, plan
    ├── retrieved_chunks, graded_chunks, reranked_chunks
    ├── response, confidence_score
    └── trace_id, thread_id

supervisor_graph:
    START
      ↓
    [plan_node]         ← Planning subgraph: intent, tool routing, clarification detection
      ↓ (route: direct | retrieve | clarify)
    [retrieval_subgraph] ← Retrieval subgraph: rewrite, multi-query, hybrid search, CRAG
      ↓
    [reranker_subgraph]  ← Reranker subgraph: score, filter to top-k, set confidence_score
      ↓
    [generation_subgraph] ← Generation subgraph: prompt assembly, LLM call, citation
      ↓
    [confidence_gate]   ← Rule-based: confidence_score < threshold → escalate/clarify
      ↓
    END
```

**Subgraph internals:**

`retrieval_subgraph`:
```
rewrite_query → [expand_queries] → retrieve → grade_docs → check_sufficient
                                                  ↑ (CRAG retry loop)
```

`reranker_subgraph`:
```
rerank → score_chunks → filter_top_k → set_confidence
```

`generation_subgraph`:
```
build_prompt → call_llm → extract_citations → format_response
```

### 5.5 Communication pattern: Command

LangGraph supervisor routing via `Command`:
```python
def supervisor_route(state: SupervisorState) -> Command:
    if state["intent"] in {"chit_chat", "out_of_scope"}:
        return Command(goto="generation_subgraph", update={"skip_retrieval": True})
    return Command(goto="retrieval_subgraph")
```

Subgraphs return `Command(goto="supervisor")` after completing, updating shared state.

## 6. Observability: LangFuse vs LangSmith

### 6.1 What we need for RAG eval

| Requirement | LangFuse | LangSmith |
|---|---|---|
| Retrieval span tracing | ✓ `@observe()` or manual spans | ✓ first-class |
| Score API (log hit@k, MRR per trace) | ✓ `langfuse.score()` | ✓ |
| ragas native integration | ✓ `ragas.integrations.langfuse` | ✗ needs custom evaluator |
| deepeval native integration | ✓ `DeepEvalCallbackHandler` | ✗ needs custom evaluator |
| Dataset + experiment runner | ✓ UI + API | ✓ richer UI |
| Annotation queues (human review) | ✗ | ✓ |
| LangGraph tracing | Via callback handler | First-class (graph state transitions) |
| Self-hostable | ✓ Docker Compose | ✗ Cloud only (enterprise for on-prem) |
| GDPR / data residency | ✓ | ✗ (no self-host without enterprise) |
| Cost | Free (self-hosted) | Per-trace pricing at scale |

### 6.2 The LangSmith case

LangSmith has genuine advantages for LangGraph-heavy projects:
- **Trace diffs**: compare graph state transitions across runs visually
- **Playground**: tweak a prompt in the UI, re-run on stored traces without code changes
- **Annotation queues**: route failed traces to human reviewers — critical for the feedback loop
- **Hub**: version and share prompts across team

These matter when:
1. A team is doing iterative prompt engineering in the UI
2. The feedback loop (thumbs up/down → label → gold dataset) is operational
3. GDPR is not a constraint (or enterprise plan is justified)

### 6.3 LangFuse first is the right call

**Why not start with LangSmith and switch:**
- The eval harness (ragas + deepeval) has native LangFuse integrations. Wiring LangSmith
  requires custom evaluator functions for both — non-trivial extra work upfront.
- Self-hosting LangFuse takes ~30 minutes with Docker Compose. LangSmith cloud costs money
  from day one, and on-prem needs an enterprise contract.
- The LangSmith advantages (annotation queues, playground) only pay off when there's a feedback
  loop to operate and a team using the UI. That's Phase 2, not MVP.
- Switching from LangFuse to LangSmith later is straightforward: replace `CallbackHandler` import,
  update env vars, done. The `get_langfuse_handler()` pattern (see listen-wiseer
  `src/utils/langfuse_tracing.py`) isolates the dependency cleanly.

**Confirmed decision: LangFuse for MVP. Revisit LangSmith when annotation queues are needed.**

### 6.4 Template observability design

LangFuse must be optional — not everyone self-hosts it, and tests must pass without it.

```python
# src/utils/tracing.py
def get_tracing_handler(session_id: str) -> CallbackHandler | None:
    """Return LangFuse handler if LANGFUSE_ENABLED=true, else None."""
    if not settings.langfuse_enabled:
        return None
    return CallbackHandler(...)
```

Activated by `LANGFUSE_ENABLED=true` env var. LangGraph `config["callbacks"]` accepts `None`
gracefully — no conditional wiring needed in graph code.

**structlog as baseline** (always on): each node logs its operation + key metrics as structured
JSON. This gives observability without LangFuse for dev and CI.

## 7. Registry Pattern: Keep for Multi-Agent Template

### 7.1 What the registry does

Decorator-based factory allowing env-var-driven strategy selection:
```python
@Registry.register("reranker", "cross_encoder")
class CrossEncoderReranker: ...

@Registry.register("reranker", "llm_listwise")
class LLMListwiseReranker: ...

# Instantiation — no code change to swap
reranker = Registry.create("reranker", settings.reranker_strategy)
```

Implemented in `src/registry.py` (help-assistant) and `src/rag_core/registry.py` (listen-wiseer).

### 7.2 Why the multi-agent design makes the registry more valuable, not less

In a multi-agent supervisor, each subgraph is instantiated with a concrete strategy. Without
a registry, swapping strategies requires changing the supervisor's wiring code. With it:

```python
# supervisor builds subgraphs from registry — no code changes for A/B tests
retrieval_subgraph = build_retrieval_subgraph(
    retriever=Registry.create("retriever", settings.retrieval_strategy),
    reranker=Registry.create("reranker", settings.reranker_strategy),
)
```

The registry is the **injection point** for all strategy variation. This is exactly the use case
it was designed for — and with 3+ swappable dimensions (chunker, retriever, reranker), it earns
its overhead.

### 7.3 Simplifications for the template

The help-assistant registry has one footgun: strategies must be imported for `@register` to fire.
Import order matters. Fix:

```python
# src/rag_template/registry.py — explicit registration at module level, no side-effect imports
from rag_template.retrieval.strategies import HybridRetriever, DenseRetriever
from rag_template.reranker.strategies import CrossEncoderReranker, LLMListwiseReranker
from rag_template.preprocessing.strategies import HtmlAwareChunker, ParentDocChunker

Registry.register("retriever", "hybrid")(HybridRetriever)
Registry.register("retriever", "dense")(DenseRetriever)
# ... etc
```

`Registry.clear()` only appears in `conftest.py` fixtures — never in application code.

**Decision: keep. Simplify registration pattern.**

## 8. Production Lean: What to Cut vs What to Keep

The template must be lean enough for real-time production (< 2s p95 latency target).

### Latency budget analysis

| Component | Latency | Can cut? |
|---|---|---|
| Query rewrite (Haiku) | ~200ms | Only if single-turn (rule-based coreference skip) |
| Retrieval (OpenSearch hybrid) | ~50–100ms | No |
| CRAG grading (Haiku, per chunk) | ~300ms (5 chunks) | Reduce k, or skip on simple queries |
| Reranker (cross-encoder) | ~50–100ms (5 chunks) | Configurable, skippable |
| Reranker (LLM listwise) | ~400–800ms | Experiment only, not prod default |
| Generation (Sonnet) | ~800–1500ms | Use streaming to hide latency |
| **Total (cross-encoder path)** | ~1.4–2.1s | Acceptable |
| **Total (LLM reranker path)** | ~2.0–3.0s | Borderline — streaming required |

### What to keep lean

- **No LLM calls in the hot path that aren't needed**: query rewrite is conditional (skip if
  single-turn, skip if no coreference signals). CRAG grading can be skipped for `chit_chat`.
- **CRAG retry is expensive**: one retry adds another retrieval + grading cycle (~400ms). Only
  fire when confidence is genuinely low. `retry_count` cap = 1 is right for prod.
- **Reranker n-candidates**: retrieve k=10, CRAG to ~5, rerank to top 3. Don't send 10 chunks
  to the LLM reranker — it'll blow the latency budget.
- **Multi-query expand**: 3 variants is the sweet spot; more adds retrieval latency with diminishing
  recall improvement.
- **Registry overhead**: negligible — one dict lookup at startup, not per-request.

### What to cut from template (not needed for prod src)

- API layer (FastAPI + SSE streaming) — out of scope per requirements
- Frontend — out of scope
- Redis checkpointer — `MemorySaver` default for template; Redis is an injection
- LangFuse integration — optional plugin, not in core
- Docker Compose — infra concern, not in `src/`

## 9. ragas vs deepeval — Eval Framework Decision

### ragas

Best for **RAG-specific batch benchmarking**:
- `faithfulness` — is the answer grounded in the retrieved context?
- `answer_relevancy` — does the answer address the query?
- `context_precision` — are the retrieved chunks actually relevant?
- `context_recall` — were the relevant chunks retrieved?
- Native LangFuse integration: `ragas.integrations.langfuse`
- Run as experiments against a golden dataset, compare runs

### deepeval

Best for **CI regression testing**:
- pytest plugin: `deepeval test run` — integrates with the test suite
- `DeepEvalCallbackHandler` for LangFuse
- Custom metrics via `GEval` (define your own evaluation criteria)
- `HallucinationMetric`, `AnswerRelevancyMetric`, `FaithfulnessMetric`

### Decision: both

- `ragas` for corpus-level quality benchmarks (run manually after corpus changes)
- `deepeval` for CI regression (run in `tests/evals/` as part of `uv run pytest`)
- Both log to LangFuse via their native handlers

## 10. Open Decisions — Groomed Backlog

### Decided (these are answers, not questions)

| # | Decision | Answer |
|---|---|---|
| D1 | Single graph vs supervisor multi-agent? | **Supervisor** — retrieval / reranker / generation as independent subgraphs |
| D3 | Rule-based or LLM-based intent classification? | **Rule-based default**, LLM via `planning_mode: "llm"` config |
| D4 | Keep or drop Registry? | **Keep**, simplify registration to explicit `__init__.py` |
| D9 | LangFuse required or optional? | **Optional plugin** — structlog baseline always on |
| Obs | LangFuse or LangSmith first? | **LangFuse first** — native ragas/deepeval, self-hostable, swap later |
| Eval | ragas or deepeval? | **Both** — ragas for experiments, deepeval for CI |

### Still open (need decision before/during build)

**D2. Cross-encoder vs LLM listwise as the default reranker?**
- Template ships both, registered under `"cross_encoder"` and `"llm_listwise"`
- **Default config**: `RERANKER_STRATEGY=cross_encoder` (lower latency, no API cost)
- LLM listwise used for eval comparison, not prod default
- **Resolution**: implement both; eval suite decides which wins; document findings

**D5. Retriever abstraction: OpenSearch-only or abstract protocol?**
- Lean: just `OpenSearchRetriever` (template targets the same prod stack)
- Flexible: `Retriever` protocol + `OpenSearchRetriever` + `InMemoryRetriever` for tests
- **Lean toward**: abstract protocol — `InMemoryRetriever` is needed anyway for unit tests
  to avoid Docker dependency

**D6. Embedder: local E5 or API-based option?**
- E5 local is the right prod choice (GDPR, no API cost, already dep)
- But `multilingual-e5-large` is 2.2GB — slow startup in tests
- **Resolution**: abstract `Embedder` protocol; `MultilingualEmbedder` (E5) for prod;
  `MockEmbedder` returning fixed-dim random vectors for unit tests

**D7. State shape: extend `RAGState` from help-assistant or clean-slate?**
- help-assistant `RAGState` is missing `reranked_chunks` and `confidence_score`
- Clean-slate is cleaner — `SupervisorState` designed for 4-agent shape
- **Lean toward**: clean-slate `SupervisorState`, document mapping from help-assistant for migration

**D8. Confidence gate: rule-based threshold or LLM self-assessment?**
- help-assistant: `confident: bool` (set heuristically, unclear how)
- Template: `confidence_score: float` from reranker, `CONFIDENCE_THRESHOLD=0.4` in config
- **Decision**: reranker sets the score, gate applies the threshold — rule-based, no extra LLM call

## 12. Architectural Context: Where the Template Fits

```
Conversational UI (Frontend)
    → TS Backend / ADK Orchestrator  ← multi-agent orchestration layer
        → Python FastAPI service     ← wraps the template
            → supervisor_graph       ← rag_template entry point
                → retrieval_subgraph → OpenSearch
                → reranker_subgraph  → cross-encoder / LLM
                → generation_subgraph → Claude Sonnet
        → [future] task agents, memory agents, etc.
```

The template is `supervisor_graph` and its subgraphs — everything below the FastAPI wrapper.
The API layer, CORS, SSE streaming, Redis checkpointer are the service layer, not the template.

## 13. Benchmarks from Production (RAPTOR v1 — sevdesk/German market)

> Source: help_center_agent_research.md + answer_quality_initiative.md
> These are real numbers from a deployed prototype, not estimates.

### RAPTOR v1 baseline (German market, all-MiniLM-L6-v2, dense-only)

| Metric | v1 Baseline | After V2 enrichment | Hybrid + reranker |
|---|---|---|---|
| Hit rate | 43% | 43% (→39%*) | **68%** |
| Precision@5 | 0.13 | 0.13 (→44%*) | **0.28** |
| MRR | — | 0.29 | — |
| Hallucination rate | **48%** | — | <5% target |

*Percentages show change vs prior version, not absolute values.

**Root causes of 48% hallucination rate:** corpus coverage gaps + no fallback behavior. Not the model.

### Retrieval strategy comparison (from RAPTOR experiments)

| Strategy | Precision@5 | Recall@5 | Hit Rate |
|---|---|---|---|
| Dense only | 0.15 | 0.18 | 45% |
| Sparse (BM25) | 0.12 | 0.25 | 50% |
| Hybrid (RRF) | 0.20 | 0.28 | 58% |
| **Hybrid + cross-encoder reranker** | **0.28** | **0.32** | **68%** |

**Key takeaway:** hybrid alone is a meaningful improvement over dense-only (+13pp hit rate).
Adding the cross-encoder reranker on top adds another 10pp. Both are worth doing.

### Reranker caveat

At small corpus sizes (<1K chunks), cross-encoder reranking can **degrade** quality by
overfitting to surface-level similarity. Test on actual corpus before committing to reranking.
The 18–42% precision improvement comes from production systems with larger corpora.

### Response style finding (RAPTOR agent-assist)

RAPTOR suggestions were ignored by human agents despite correct technical content because:
- Suggestions were 3× the length of agent responses
- Hedged language: *"könnte möglicherweise"*, *"Überprüfe bitte..."*
- Agents wrote direct responses and ignored suggestions entirely

**Implication for generation prompts:**
- System prompt must enforce direct, actionable language
- Max response length should be configured (not open-ended)
- No hedging phrases — if confidence is low, route to confidence_gate, don't hedge in the answer

### BGE-M3 vs multilingual-e5-large

The research doc uses `multilingual-e5-large` (already in help-assistant). BGE-M3 is flagged as
the better multilingual open-source model:

| Model | Multilingual | Self-hosted | Notes |
|---|---|---|---|
| `all-MiniLM-L6-v2` | No | Yes | Silent fails on non-English — never use for Danish |
| `multilingual-e5-large` | Yes (100 langs) | Yes (GPU) | Proven in help-assistant |
| `BGE-M3` | Yes (100 langs) | Yes (GPU) | Claimed better multilingual; not yet benchmarked |
| `Cohere embed-v4` | Yes (100+ langs) | No (API) | Best if no GPU; data leaves infra |

Template uses E5-large (already integrated, proven). BGE-M3 is a registered alternative once
benchmarked. `all-MiniLM-L6-v2` is explicitly excluded from registration.

### BM25 Danish stemmer

BM25 with default (English) stemmer degrades on Danish morphology. The `bm25s` library supports
language-specific stemmers but requires explicit configuration:

```python
# Must configure Danish stemmer — default is English, which silent-fails on Danish
from bm25s.tokenization import Tokenizer
tokenizer = Tokenizer(stemmer=Stemmer("danish"))
```

This applies to the `OpenSearchRetriever` BM25 configuration and any standalone BM25 implementation.
Not needed for `InMemoryRetriever` in tests (test corpus is English).

## 14. Production Metadata Schema (from answer_quality_initiative.md)

The production deployment (Danish help-assistant) requires richer chunk metadata than the
current `ChunkMetadata` schema. These fields are needed for personalised retrieval
(market filtering, plan-tier filtering) and freshness tracking.

### Required production fields

```python
class ChunkMetadata(BaseModel):
    # Current (template minimum)
    url: str
    title: str
    section: str | None = None
    language: str = "da"
    doc_id: str
    parent_id: str | None = None

    # Production additions (help-assistant v2+)
    market: str | None = None            # "dk", "de", "nl" etc — for market filtering
    product_area: str | None = None      # "billing", "tax_filing", "integrations" etc
    content_type: str | None = None      # "faq", "guide", "policy", "blog"
    plan_tier: str | None = None         # "starter", "pro", "enterprise" — access control
    last_updated: str | None = None      # ISO 8601 — freshness detection
    source_id: str | None = None         # original CMS/Intercom article ID
    completeness_score: float | None = None  # quality gate at ingestion (0–1)
```

**Template approach:** `ChunkMetadata` ships with the minimum set. Production fields are
additive (all `None` by default) — no migration needed when adding them later.

### Metadata filtering at retrieval time

The production API accepts user context (market, language, plan_tier) injected by the BE
from the authenticated session. This filters at ANN query time, not post-retrieval:

```python
# Retriever.search() accepts metadata_filter
await retriever.search(
    query_text=query,
    query_vector=embedding,
    k=10,
    metadata_filter={"market": "dk", "language": "da", "plan_tier": "pro"},
)
```

`InMemoryRetriever` must support this filter interface for unit tests — even if it's just
a simple dict match on chunk metadata fields.
